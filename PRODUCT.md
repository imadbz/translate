# Translate — Product Direction

## Core Philosophy

**The developer never thinks about i18n.** They write `<button>Pay now</button>` and that's it. No `t()` calls, no key files, no translation imports, no wrapping components. The plugin handles everything at build time. The developer's source files are read-only inputs — they never change.

**Removing the plugin is zero-risk.** Delete one line from `vite.config.ts` and the app works in English. No broken imports, no missing functions, no cleanup. This is the trust signal for first install.

## Architecture Decisions

### Server-side extraction, not local AST heuristics

Algorithmic AST heuristics produce too many false positives. Our differentiator is LLM-powered extraction on the server. The Vite plugin is a thin client — it uploads source files, the server does the thinking, and returns transformed code.

The server is the **single source of truth** for:
- Which strings are translatable
- What keys they map to
- Which locales are enabled for a project
- The translation content for each locale

### `t()` calls are an internal intermediate representation

The plugin inserts `t()` calls into the build output, but the developer never writes or sees them. These calls resolve at runtime against the active locale's translation map. This enables runtime language switching without per-locale builds.

### Runtime language switching, not per-locale builds

One build serves all locales. The `@translate/react` package provides:
- `TranslateProvider` — wraps the app (auto-injected by the plugin)
- `useLocale()` — `{ locale, setLocale, availableLocales }`
- `useTranslation()` — returns the `t()` function

Language detection priority:
1. Explicit user choice (persisted in localStorage)
2. Browser preference (`navigator.languages` / `Accept-Language` header)
3. Default locale fallback

### Translations live on disk, managed by the server

The `translations/` directory is infrastructure, not source code. The developer never manually edits it.

- **`en.json`** — written by the plugin after each build (extracted strings)
- **`{locale}.json`** — written by the server / GitHub Action
- All locale files are committed to git for version control and CI reproducibility

The plugin reads all `translations/*.json` via a virtual module and bundles them. The server decides which locales are enabled per project.

### The server owns locale configuration

When the plugin uploads files, the server knows (via `projectId`) which locales are enabled. The server returns:
- Transformed source files (with `t()` calls)
- Translation maps for **all enabled locales**

The plugin writes these to `translations/` and the virtual module picks them up. The developer never configures locale lists.

## What the developer's repo looks like

```
src/
  App.tsx                    ← plain React, no i18n awareness
  CheckoutPage.tsx           ← "Pay now" forever, never changes
translations/
  en.json                    ← auto-generated by plugin
  fr.json                    ← auto-generated by server/action
  de.json                    ← auto-generated by server/action
vite.config.ts               ← one plugin line: translate({ serverUrl })
```

## What the developer experiences

- Writes normal React, no i18n awareness
- Git history is clean — source files never touched
- Removes plugin → app works in English, zero breakage
- Language switcher works out of the box via `useLocale()` hook
- New locales appear automatically when the server enables them

## Future: GitHub Action

A GitHub Action that:
1. Triggers when `en.json` changes (new/updated strings detected)
2. Calls the translation API for all enabled locales
3. Commits updated locale JSON files back to the repo
4. PR-based workflow so the team can review translation changes

This is deferred — the plugin works fully without it by reading whatever locale files exist on disk.

## TODO: LLM Rate Limit Handling

Currently files are processed sequentially to respect API rate limits (e.g. 5 req/min on Anthropic free tier). This is not acceptable for production — a 50-file project would take minutes.

Possible solutions:
- **Batch multiple files into a single LLM call** — send 5-10 files per prompt, reducing total requests
- **Request queue with rate-aware backoff** — track rate limit headers, schedule requests to stay under the limit
- **Higher rate limit tier** — production server would have a paid plan with much higher limits
- **Hybrid approach** — use AST for simple cases (obvious JSX text), LLM only for ambiguous strings
- **Persistent cache** — only re-process files that changed since last build (hash-based, stored on server)

This is a critical path issue that must be solved before production use.

## Non-goals

- **No per-locale builds.** One build, all locales bundled.
- **No developer-facing key management.** Keys are generated and managed by the server.
- **No manual translation editing.** The `translations/` directory is machine-managed.
- **No framework lock-in for the server.** The AST extraction is a placeholder — the real server uses LLM. The protocol (upload files → get back transforms + translations) is framework-agnostic.
